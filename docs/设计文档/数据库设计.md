## 一、设计原则与规范

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| **第三范式** | 表设计遵循第三范式，减少数据冗余 |
| **适当反范化** | 对于查询频繁的关联数据，可适当冗余存储 |
| **统一字段** | 所有表包含统一的基础字段（创建时间、更新时间等） |
| **逻辑删除** | 使用 `delete_flag` 字段实现逻辑删除，保留数据可追溯性 |
| **大字段分离** | JSON、TEXT等大字段单独存放或使用扩展表 |

### 1.2 基础字段规范

所有业务表必须包含以下基础字段：

| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `id` | VARCHAR(32) | - | 主键，雪花算法生成 |
| `create_by` | VARCHAR(100) | NULL | 创建人 |
| `create_time` | DATETIME(3) | NULL | 创建时间 |
| `update_by` | VARCHAR(100) | NULL | 更新人 |
| `update_time` | DATETIME(3) | NULL | 更新时间 |
| `update_count` | INT | 0 | 更新次数，乐观锁使用 |
| `delete_flag` | CHAR(1) | '0' | 删除标志：0-未删除，1-已删除 |
| `delete_time` | DATETIME(3) | NULL | 删除时间 |

### 1.3 字段类型规范

| 数据类型 | Java类型 | 使用场景 |
|----------|----------|----------|
| VARCHAR(32) | String | 主键ID、短编码 |
| VARCHAR(100) | String | 名称、编码、创建人等 |
| VARCHAR(255) | String | 普通文本、描述 |
| VARCHAR(500) | String | 较长文本 |
| VARCHAR(2000) | String | URL、路径等 |
| TEXT | String | 大文本、配置内容 |
| JSON | String/Object | JSON格式数据，MySQL 5.7+ |
| INT | Integer | 整数、状态码、计数 |
| BIGINT | Long | 大整数、数量统计 |
| DECIMAL(18,4) | BigDecimal | 金额、精确小数 |
| DATETIME(3) | LocalDateTime | 日期时间（含毫秒） |
| DATE | LocalDate | 仅日期 |
| TINYINT | Byte/Boolean | 布尔值、小范围枚举 |

## 二、命名规范

### 2.1 表命名规范

格式：`{模块前缀}_{业务名称}`

| 模块 | 前缀 | 示例 |
|------|------|------|
| 数据源管理 | ds | ds_datasource_info |
| 数据集成 | di | di_data_pipeline |
| 数据标准 | dn | dn_dataset |
| 主数据 | md | md_organization |
| 元数据 | meta | meta_model |
| 数据质量 | dq | dq_rule |
| 数据资产 | da | da_asset |
| 数据安全 | ds_sec | ds_sec_mask_rule |
| HIE集成 | hie | hie_interaction_service |

### 2.2 字段命名规范

- 全部小写，单词间使用下划线分隔
- 避免使用数据库保留字
- 布尔字段以 `is_` 开头，如 `is_active`
- 外键字段以 `_id` 结尾，如 `datasource_id`
- 状态字段使用 `status`，枚举值使用英文大写

### 2.3 索引命名规范

| 索引类型 | 命名格式 | 示例 |
|----------|----------|------|
| 主键 | PK_{表名} | PK_ds_datasource_info |
| 唯一索引 | UK_{表名}_{字段} | UK_ds_datasource_info_code |
| 普通索引 | IDX_{表名}_{字段} | IDX_di_data_pipeline_project_id |
| 组合索引 | IDX_{表名}_{字段1}_{字段2} | IDX_md_practitioner_org_id_dept_id |

## 三、索引设计规范

### 3.1 索引设计原则

1. **主键必须建立**：所有表必须有主键，使用雪花算法生成
2. **外键建索引**：所有外键字段建立索引
3. **查询条件建索引**：WHERE条件中的高频字段建立索引
4. **排序字段建索引**：ORDER BY字段建立索引
5. **组合索引字段顺序**：等值查询字段在前，范围查询字段在后
6. **控制索引数量**：单表索引不超过5个
7. **避免冗余索引**：不建立重复或包含关系的索引

### 3.2 索引示例

```sql
-- 主键索引
PRIMARY KEY (`id`)

-- 唯一索引
UNIQUE KEY `UK_ds_datasource_info_code` (`datasource_code`)

-- 普通索引
KEY `IDX_di_data_pipeline_project_id` (`project_id`)

-- 组合索引
KEY `IDX_md_practitioner_org_id_dept_id` (`org_id`, `dept_id`)
```

## 四、Flyway版本管理

### 4.1 目录结构

```
datasphere-database/
└── src/main/resources/db/
    ├── migration/
    │   ├── V1.0.0__init_schema.sql           # 初始化表结构
    │   ├── V1.0.1__init_data.sql             # 初始化数据
    │   ├── V1.1.0__add_quality_module.sql    # 数据质量模块
    │   ├── V1.1.1__add_asset_module.sql      # 数据资产模块
    │   └── V1.1.2__add_security_module.sql   # 数据安全模块
    └── baseline/
        └── V1.0.0__baseline.sql              # 基线脚本(已有数据)
```

### 4.2 版本命名规范

```
V{主版本}.{次版本}.{补丁版本}__{描述}.sql
```

| 版本段 | 说明 | 示例 |
|--------|------|------|
| 主版本 | 大功能模块变更 | 1.x.x -> 2.x.x |
| 次版本 | 功能增强、新模块 | 1.0.x -> 1.1.x |
| 补丁版本 | Bug修复、小优化 | 1.0.0 -> 1.0.1 |

### 4.3 脚本编写规范

1. **可重复执行**：脚本必须可重复执行，使用 `IF NOT EXISTS` 或 `DROP IF EXISTS`
2. **事务控制**：DDL语句使用事务包裹
3. **注释说明**：每个脚本头部注明版本、日期、作者、变更说明
4. **数据脚本分离**：表结构变更和数据初始化分离到不同脚本
