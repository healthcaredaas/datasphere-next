## 二、基础框架选型

### 2.1 Spring Boot 3.5.11

**选型理由**:

| 优势 | 说明 |
|------|------|
| **长期支持** | Spring Boot 3.x 是长期支持版本，提供到2026年的支持 |
| **性能提升** | 基于Spring Framework 6.x，启动速度提升20%以上 |
| **GraalVM支持** | 原生镜像支持，适合云原生部署 |
| **JDK 21支持** | 支持虚拟线程(Virtual Threads)，提升并发性能 |
| **安全增强** | 默认安全策略增强，依赖漏洞修复 |

**关键变更**:
- 最低JDK版本要求：JDK 17
- Jakarta EE命名空间迁移：`javax.*` -> `jakarta.*`
- 自动配置优化
- Actuator端点改进

### 2.2 JDK 21 (LTS)

**选型理由**:

| 特性 | 说明 |
|------|------|
| **虚拟线程** | Project Loom正式特性，轻量级并发模型 |
| **模式匹配** | switch表达式增强，代码更简洁 |
| **Record模式** | Record类的模式匹配，解构数据更便捷 |
| **性能优化** | G1/ZGC垃圾收集器性能提升 |
| **LTS版本** | 长期支持版本，稳定性有保障 |

**选型理由**:

### 3.2 Spring Cloud Alibaba 2025.0.0

**选型理由**:

| 优势 | 说明 |
|------|------|
| **国产生态** | 阿里巴巴开源，中文社区活跃 |

| 组件 | 版本 | 用途 |

| RocketMQ | 5.3.x | 消息队列 |
| Dubbo | 3.3.x | RPC框架 |

### 3.3 Apache Dubbo 3.3.x

**与OpenFeign对比**:

| 对比项 | Dubbo | OpenFeign |
|--------|-------|-----------|
| **协议** | 自定义Dubbo协议/HTTP/Triple | HTTP |
| **性能** | 更高，二进制序列化 | 较低，JSON序列化 |
| **服务治理** | 原生支持 | 依赖Ribbon/Hystrix |
| **跨语言** | Triple协议支持gRPC | HTTP通用 |
| **生态** | 阿里生态 | Spring生态 |

**选型结论**: 选用Dubbo 3.3.x，原因：
1. 与Nacos深度集成
2. Triple协议支持，可扩展性强
3. 服务治理能力更强
4. 医疗行业对性能要求较高

## 四、数据存储选型

### 4.1 MySQL 8.0+

**选型理由**:

| 优势 | 说明 |
|------|------|
| **性能提升** | 比5.7版本性能提升2倍 |
| **窗口函数** | 复杂查询支持更好 |
| **JSON支持** | 原生JSON类型，灵活存储 |
| **CTE** | 公用表表达式，递归查询 |
| **安全性** | 默认认证插件升级 |

**配置建议**:

```ini
# my.cnf 推荐配置
[mysqld]
# 字符集
character-set-server=utf8mb4
collation-server=utf8mb4_unicode_ci

# InnoDB配置
innodb_buffer_pool_size=4G
innodb_log_file_size=1G
innodb_flush_log_at_trx_commit=2

# 连接配置
max_connections=500
max_connect_errors=1000

# 查询缓存
query_cache_type=1
query_cache_size=256M
```

### 4.2 Redis 7.x

**选型理由**:

| 特性 | 说明 |
|------|------|
| **性能** | 10万+ QPS |
| **数据结构** | String/Hash/List/Set/ZSet/Stream |
| **持久化** | RDB + AOF |
| **集群** | Redis Cluster原生支持 |
| **新特性** | Functions、ACL改进、多部分AOF |

**使用场景**:

| 场景 | 数据类型 | 说明 |

| 限流计数 | String | 滑动窗口限流 |

### 4.3 Elasticsearch 8.x (可选)

## 五、数据集成引擎选型

### 5.1 Apache SeaTunnel 2.3.x

**选型理由**:

| 优势 | 说明 |
|------|------|
| **活跃社区** | Apache顶级项目，持续迭代 |
| **多引擎支持** | 支持Spark/Flink/自研Zeta引擎 |
| **连接器丰富** | 100+连接器，支持主流数据源 |
| **性能优秀** | 比DataX快30%以上 |
| **CDC支持** | 支持实时数据同步 |

**支持的连接器**:

| 类型 | 连接器 |
|------|--------|
| 关系型数据库 | MySQL、Oracle、PostgreSQL、SQLServer、达梦、人大金仓 |
| 大数据存储 | Hive、ClickHouse、StarRocks、Doris |
| 消息队列 | Kafka、RocketMQ、Pulsar |
| 文件存储 | HDFS、S3、FTP、LocalFile |
| NoSQL | MongoDB、Redis、Elasticsearch |

### 5.2 Debezium 2.3.x (CDC)

**选型理由**:

| 优势 | 说明 |
|------|------|
| **CDC标准** | CDC领域事实标准 |
| **低延迟** | 秒级数据捕获 |
| **一致性** | 基于数据库日志，保证一致性 |
| **多数据库** | 支持MySQL、PostgreSQL、Oracle、SQLServer |

**架构设计**:

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Source    │────▶│   Debezium  │────▶│   Kafka/    │────▶│    Sink     │
│  Database   │     │  Connector  │     │  RocketMQ   │     │  Database   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
                           │
                           │ 捕获变更事件
                           ▼
                    ┌─────────────┐
                    │  Offset存储  │
                    │  (记录位置)  │
                    └─────────────┘
```

| 对比项 | SeaTunnel | DataX | Flink CDC |
|--------|-----------|-------|-----------|
| **社区活跃度** | ★★★★★ | ★★★☆☆ | ★★★★☆ |

## 六、消息队列选型

### 6.1 Apache RocketMQ 5.3.x

**选型理由**:

| 优势 | 说明 |
|------|------|

## 九、技术风险评估

### 9.1 风险矩阵

| 风险项 | 概率 | 影响 | 风险等级 | 应对措施 |
|--------|------|------|----------|----------|
| Spring Boot 3.x迁移 | 高 | 高 | **高** | 制定详细迁移计划，逐步验证 |
| Jakarta包名变更 | 高 | 中 | **中** | 使用IDE批量替换，测试覆盖 |
| 多租户移除 | 中 | 中 | **中** | 评估数据隔离需求，设计替代方案 |
| SeaTunnel兼容性 | 低 | 高 | **中** | 提前搭建测试环境验证 |
| 性能回归 | 中 | 中 | **中** | 建立性能基线，持续监控 |
| 第三方组件弃用 | 低 | 中 | **低** | 持续关注社区动态 |

### 9.2 缓解措施

#### 9.2.1 Spring Boot 3.x迁移风险

**风险描述**: 代码改动量大，可能存在不兼容的API变更

**缓解措施**:
1. 使用Spring Boot Migrator工具自动检测
2. 建立feature分支，小步快跑
3. 编写自动化测试脚本，快速回归
4. 保留旧版本环境用于对比验证

#### 9.2.2 Jakarta包名变更风险

**风险描述**: 大量import语句需要修改，容易遗漏

**缓解措施**:
1. 使用IDE全局替换功能
2. 编写脚本自动检测`javax.*`残留
3. 编译时开启严格模式
4. 代码审查重点检查

#### 9.2.3 多租户移除风险

**风险描述**: 移除多租户可能影响数据隔离，需要评估业务影响

**缓解措施**:
1. 评估当前多租户使用场景
2. 设计替代隔离方案（如按应用/项目隔离）
3. 数据迁移时保留tenant_id字段作为冗余
4. 与业务部门确认需求
